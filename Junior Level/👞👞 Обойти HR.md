# 1. Методы HTTP 

1. GET
2. **HEAD** — запрашивает ресурс так же, как и метод `GET`, но без тела ответа.
3. PUT — заменяет все текущие представления ресурса данными запроса.
4. **PATCH** — предназначен для частичного обновления ресурса. В отличие от PUT, PATCH изменяет только указанные поля, оставляя остальные нетронутыми.
5. POST
6. DELETE
7. **OPTION** — для получения информации о поддерживаемых методах.
8. **CONNECT** — обычно используется прокси-серверами для установления прямой связи с целью по HTTP-соединению.
9. **TRACE** — используется для получения диагностических данных от сервера. Позволяет клиенту проверить, как сервер видит запрос, поскольку сервер должен вернуть клиенту точную копию запроса, который был отправлен серверу.

# 2. Разница между GET и POST

GET нужен для запроса данных, POST для отправки данных. Итоговое назначение методов полагается на RFC, REST, SOAP и RPC, но абсолютного, строго технического требования нет.

> **RFC (Request for Comments)** — **серия документов, описывающих технические аспекты и стандарты интернета**.

> Фактические GET тоже может отправлять данные, потому что RFC стандарты не содержат абсолютного запрета на тело запроса в методе GET. Благодаря этому, например, в HTTP/1 возможна контрабанда запросов.

# 3. Что можно получить из LFI

LFI - **априори RCE** через различные манипуляции:
1. Подключение интерпретатора на выполнение заинклюженых файлов;
2. Подключение кэшей либо сессионных идентификаторов;
3. Метод через краш PHP различных версий;
4. Метод через PHP info, когда есть путь к temp файлу;
5. Методики через Race conditions, создающие кэш уровня PHP fpm и ты в моменте до его удаления инклудишь и выполняешь RCE;
6. Если инфраструктура не разделена на разные хосты/контейнеры, то условно на сайте возможно загрузить картинку с php-shell и заинклудить его.

**DoS:**
> У PHP уровень абстракции потоков завязан на сессионный идентификатор. Каждая сессия для PHP - это отдельный mutex. Ты можешь безгранично читать `/dev/urandom`, он начнёт падать. Но тебе необходимо будет сгенерировать большое количество сессий.

# 4. Где могут хранится хэши паролей в Linux кроме `/etc/shadow`

1. В `/etc/passwd` - идеологически могут храниться здесь;
2. PAM - это библиотека, которые отвечают за аутентификацию в операционной системе. PAM используется везде, где требуется аутентификация пользователя или проверка его прав, например, при подключении через SSH, FTP, или, например ldap. Хэши могут храниться в кэше ldap.
3. Программист может хранить хэши **хоть где**, даже вовне - на github.
4. В оперативной памяти.

# 5. Чем опасен SUID bit

1. Локальным повышением привилегий.
   > Если исполняемый файл может выполнять системные вызовы, например, execve (заменяет текущий образ процесса программой из файла). Например nmap может выполнять код. Здесь полезна тулза **gtfobins.github.io**.
2. Деструктивные действия: удаление директорий и т.д.
